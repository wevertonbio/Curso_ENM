RuHere

Obtenção e limpeza de dados de ocorrência usando informações de especialistas

Author

Weverton Trindade & Fernanda S. Caron

Introdução

Dados de ocorrência de espécies são amplamente utilizados em estudos de modelagem de nicho ecológico, macroecologia, biogeografia, conservação da biodiversidade, etc.

Nas últimas décadas, a disponibilidade desses dados aumentou enormemente graças a repositórios como GBIF, SpeciesLink, BIEN e iDigBio.

No entanto, esses dados frequentemente apresentam problemas importantes de qualidade, como:

  • coordenadas inválidas ou imprecisas;
  • inconsistências entre coordenadas e metadados;
  • registros duplicados;
  • ocorrências fora da distribuição natural da espécie;
  • viés amostral espacial e ambiental.

Além disso, preparar dados de ocorrência costuma ser uma das etapas mais demoradas e subjetivas de um estudo.

O RuHere foi desenvolvido para integrar diferentes etapas de obtenção, validação e limpeza de dadosde ocorrência em um único workflow reproduzível dentro do R. Registros potencialmente problemáticos são marcados (flagged) e posteriormente removidos, permitindo maior transparência e controle por parte do usuário.

Workflow do RuHere:

  1. Obtenção de ocorrência das espécies
    Download e integração de registros de diferentes repositórios.

  2. Padronização e harmonização dos dados
    Organização de colunas e padronização espacial.

  3. Validação usando metadados
    Identificação de inconsistências espaciais e problemas nos registros.

  4. Validação usando informações de especialistas
    Comparação com distribuições conhecidas das espécies.

  5. Exploração visual e remoção de registros problemáticos
    Resumo, inspeção e filtragem das flags.

  6. Exploração e redução do viés amostral
    Thinning geográfico e ambiental.

Nesta aula, vamos percorrer esse workflow usando um exemplo simples com Araucaria angustifolia.

0. Preparando o ambiente

Antes de começarmos, vamos instalar os pacotes necessários e organizar os diretórios que serão utilizados ao longo da aula.

O RuHere está disponível no CRAN e pode ser instalado como qualquer outro pacote do R:

install.packages("RuHere")

Ao longo da aula, alguns arquivos serão baixados, incluindo dados de ocorrência e bases com distribuições de especialistas.

Vamos definir um diretório específico para armazenar esses arquivos:

data_dir <- "~/Documents/lab/RuHere/data"

1. Obtendo dados de ocorrência

O RuHere permite obter registros diretamente de diferentes repositórios de biodiversidade, incluindo GBIF, SpeciesLink, BIEN e iDigBio.

Nesta aula, vamos focar nos procedimentos de limpeza e validação. Para isso, vamos primeiramente ver como os dados podem ser obtidos.

No caso do GBIF, primeiro configuramos as credenciais de acesso. Isso precisa ser feito apenas uma vez:

library(RuHere)

## Aqui, você deve substituir cada variável com seus dados
set_gbif_credentials(
  gbif_username = "my_username",
  gbif_email = "my_email@example.com",
  gbif_password = "my_password"
)

Depois, preparamos a requisição para download, escolhendo o grupo taxonômico:

gbif_prepared <- prepare_gbif_download(
  species = "Araucaria angustifolia"
)

Em seguida, submetemos a requisição ao GBIF:

gbif_requested <- request_gbif(
  gbif_info = gbif_prepared
)

Como o download é processado pelo próprio servidor do GBIF, podemos acompanhar seu progresso:

1.1. Outros conjuntos de dados

O mesmo procedimento pode ser aplicado a outras bases de dados suportadas pelo pacote:

occ_idigbio <- get_idigbio(
  species = "Araucaria angustifolia"
)

)

Além disso, também podemos obter os dados do SpeciesLink, que também há a necessidade de configurar as credenciais associadas à sua conta na plataforma:

set_specieslink_credentials(
  "my_specieslink_key"
)

occ_specieslink <- get_specieslink(
  species = "Araucaria angustifolia"
)

2. Padronizando e integrando dados

Depois de obter registros de ocorrência, um dos principais desafios é lidar com a heterogeneidade entre bases de dados.

Cada repositório pode utilizar nomes de colunas diferentes, formatos distintos para datas e coordenadas, convenções diferentes para metadados, etc.

Por exemplo, a mesma informação pode aparecer com nomes diferentes dependendo da fonte:

Variável Exemplo
espécie species, scientificName
latitude decimalLatitude, lat
longitude decimalLongitude, lon
país country, countryCode

Para resolver isso, o RuHere utiliza o padrão Darwin Core (DwC), amplamente usado para padronização de dados de biodiversidade.

2.1. Formatando colunas

A função format_columns() harmoniza nomes de colunas, classes dos dados e codificação de texto:

occ_gbif_standardized <- format_columns(
  occ = occ_gbif,
  metadata = "gbif"
)

occ_specieslink_standardized <- format_columns(
  occ = occ_specieslink,
  metadata = "specieslink"
)

occ_bien_standardized <- format_columns(
  occ = occ_bien,
  metadata = "bien"
)

occ_idigbio_standardized <- format_columns(
  occ = occ_idigbio,
  metadata = "idigbio"
)

Depois da padronização, os conjuntos de dados podem ser combinados.

2.2. Integrando múltiplas fontes

Depois, os dados podem ser combinados em uma única tabela usando bind_here():

occ <- bind_here(
  occ_gbif_standardized,
  occ_specieslink_standardized,
  occ_bien_standardized,
  occ_idigbio_standardized
)

3. Validação usando metadados

Depois de integrar diferentes bases de dados, o próximo passo é identificar problemas espaciais básicos.

Mesmo quando os registros parecem corretos, é comum encontrarmos coordenadas inválidas, latitude e longitude invertidas, coordenadas incompatíveis com país ou estado informados, registros próximos a capitais, instituições biológicas ou centroids administrativos ou inconsistências geradas durante a digitalização dos dados.

Um ponto importante do RuHere é que ele não remove registros automaticamente.

Em vez disso, ele adiciona flags lógicas (TRUE/FALSE) indicando se cada registro passou ou falhou em determinada verificação.

3.1. Padronizando países e estados

Antes das verificações espaciais, precisamos garantir que os metadados estejam consistentes.

Bases diferentes frequentemente utilizam grafias distintas para o mesmo local (Brasil, Brazil, BR, etc.), além de abreviações ou campos ausentes.

Primeiro, padronizamos países:

occ_standardized <- standardize_countries(
  occ = occ,
  country_column = "country",
  lookup_na_country = TRUE,
  return_dictionary = FALSE
)

A função pode inclusive inferir países ausentes a partir das coordenadas.

Vamos comparar alguns valores:

table(occ_standardized$country) |> head()
#> 
#>  [italie]        AR argentina        BI        BO   bolivia 
#>         1       160        19         1         6        12

table(occ_standardized$country_suggested) |> head()
#> 
#> argentina   bolivia    brazil   burundi     chile     china 
#>       179        18      5223         1        28         1

O mesmo pode ser feito para estados e províncias:

occ_standardized <- standardize_states(
  occ = occ_standardized,
  state_column = "stateProvince",
  country_column = "country_suggested",
  lookup_na_state = TRUE,
  return_dictionary = FALSE
)

Agora nossos dados estão mais consistentes e prontos para as primeiras verificações de qualidade espacial.

3.2. Coordenadas problemáticas

O próximo passo é identificar registros com coordenadas inválidas.

A função remove_invalid_coordinates() adiciona uma flag indicando registros com coordenadas ausentes, valores impossíveis, `NA`s ou valores vazios:

occ_standardized <- remove_invalid_coordinates(occ_standardized)

3.3. Verificando inconsistências espaciais

Mesmo com coordenadas válidas, ainda podem existir inconsistências.

Por exemplo, um registro informado como Brasil, mas cujas coordenadas caem na Argentina.

Primeiro verificamos se as coordenadas correspondem ao país informado:

occ_standardized <- check_countries(
  occ = occ_standardized$valid,
  country_column = "country_suggested"
)

Quantos registros apresentaram inconsistências?

#> FALSE  TRUE

3.4. Integrando flags do CoordinateCleaner

Além das verificações implementadas no próprio RuHere, também podemos incorporar testes do pacote CoordinateCleaner, amplamente utilizado em estudos de modelagem de nicho ecológico.

Essas funções ajudam a identificar registros suspeitos próximos de capitais, centróides administrativos, instituições biológicas, áreas urbanas e coordenadas potencialmente problemáticas.

library(CoordinateCleaner)

occ_standardized <- clean_coordinates(
  x = occ_standardized,
  lon = "decimalLongitude",
  lat = "decimalLatitude",
  species = "species"
)

Como tanto o RuHere quanto o CoordinateCleaner utilizam um sistema de flags, seus resultados podem ser integrados facilmente em um mesmo workflow.

3.5. Explorando os registros no mapa

Depois das verificações, podemos visualizar os registros espacialmente.

Muitas vezes, problemas se tornam muito mais evidentes no mapa do que em tabelas.

Por exemplo:

ggmap_here(
  occ_standardized,
  species = "Araucaria angustifolia"
)

Ou, alternativamente, podemos usar um mapa interativo:

map_here(occ_standardized)

A inspeção visual é uma etapa importante do workflow.

Ela ajuda a decidir quais flags realmente devem ser consideradas na limpeza final e evita remoções excessivamente automáticas.

Até aqui, identificamos inconsistências espaciais usando coordenadas e metadados.

Mas um registro ainda pode parecer correto e, mesmo assim, estar fora da distribuição biologicamente esperada da espécie.

É exatamente aí que entram as informações derivadas de especialistas.

4. Validação usando dados de especialistas

Até agora, utilizamos coordenadas, metadados e informações associadas aos registros. Mas um registro pode ainda assim parecer perfeitamente válido e ocorrer em um local biologicamente improvável. Por exemplo, uma espécie endêmica do sul do Brasil aparecendo no Nordeste.

Nesses casos, informações de distribuição conhecidas por especialistas podem ser extremamente úteis.

Esse é um dos principais diferenciais do RuHere.

O pacote permite comparar ocorrências com distribuições especialistas provenientes de diferentes bases de dados.

O RuHere atualmente integra informações provenientes de WCVP (World Checklist of Vascular Plants), IUCN, BIEN, Flora e Funga do Brasil e Catálogo Taxonômico da Fauna do Brasil.

4.1. Baixando bases especialistas

Essas bases precisam ser baixadas apenas uma vez e podem ser reutilizadas em análises futuras.

Como Araucaria angustifolia é uma planta nativa do Brasil, podemos verificar se os registros estão dentro da distribuição esperada segundo a Flora e Funga do Brasil.

flora_br <- florabr_here(data_dir = data_dir)

occ_flagged <- flag_florabr(
  data_dir = data_dir,
  occ = occ_standardized
)

Resumo:

table(occ_flagged$florabr_flag)
#> 
#> FALSE  TRUE 
#>  4339   772

Também podemos utilizar a distribuição descrita pela WCVP:

wcvp_br <- wcvp_here(data_dir = data_dir)

occ_flagged <- flag_wcvp(
  data_dir = data_dir,
  occ = occ_flagged
)
table(occ_flagged$wcvp_flag)
#> 
#> FALSE  TRUE 
#>   457  4654

4.2. Consenso entre múltiplas bases

Uma vantagem do RuHere é a possibilidade de combinar múltiplas fontes de informação.

Por exemplo, podemos considerar um registro válido se ele for aceito por pelo menos uma base, ou exigir concordância entre todas:

occ_flagged <- flag_consensus(
  occ = occ_flagged,
  flags = c("florabr", "wcvp"),
  consensus_rule = "all_true"
)

5. Visualizando e removendo registros marcados

Até aqui, identificamos potenciais problemas nos dados usando inconsistências espaciais, metadados e distribuições conhecidas por especialistas.

Antes de prosseguir para análises ecológicas, é importante explorar os resultados e decidir quais registros remover.

5.1. Resumindo as flags

Podemos observar quantos registros foram marcados por cada procedimento:

summary_plot <- summarize_flags(occ_flagged)

summary_plot$plot_summary

5.2. Visualizando registros

Também podemos explorar visualmente os registros no espaço geográfico:

ggmap_here(occ_flagged)

Uma das vantagens do RuHere é justamente permitir uma inspeção iterativa dos dados antes de qualquer remoção:

map_here(occ_flagged)

5.3. Removendo registros problemáticos

Depois da inspeção, podemos remover registros marcados.

Por padrão, remove_flagged() considera todas as flags presentes no conjunto de dados:

occ_clean <- remove_flagged(occ_flagged)

Vamos comparar o tamanho dos conjuntos:

nrow(occ_flagged)
#> [1] 5111

nrow(occ_clean)
#> [1] 326

Agora temos um conjunto de dados potencialmente mais adequado para análises posteriores.

6. Exploração e redução do viés amostral

Mesmo após remover registros problemáticos, ainda pode existir um problema importante: os registros nem sempre representam igualmente o espaço geográfico ou ambiental.

Em dados de biodiversidade, é comum encontrarmos regiões intensamente amostradas e outras pouco representadas. Isso geralmente acontece por fatores como proximidade de centros urbanos, facilidade de acesso, maior esforço de coleta em determinadas regiões ou preferência de coletores.

Como consequência, conjuntos de dados de ocorrência frequentemente apresentam viés amostral.

Esse tipo de viés pode influenciar fortemente modelagens de nicho ecológico e outras análises espaciais.

O objetivo é reduzir redundância espacial e ambiental.

6.1. Explorando a distribuição espacial dos registros

Antes de qualquer filtragem, vale observar como os registros estão distribuídos no espaço.

Podemos visualizar a densidade de registros usando kernel density estimation (KDE):

heatmap <- spatial_kde(occ_clean, resolution = 0.25, buffer_extent = 50, radius = 2)

ggmap_here(occ = occ_clean, size_points = 0.5, heatmap = heatmap)

6.2. Thinning geográfico

Uma estratégia comum para reduzir autocorrelação espacial é remover registros muito próximos.

No RuHere, isso pode ser feito usando thin_geo().

Por exemplo, podemos manter apenas um registro dentro de um raio de 10 km:

occ_clean_thin <- thin_geo(
  occ = occ_clean,
  d = 10
)

Vamos verificar quantos registros foram mantidos:

table(occ_clean_thin$thin_geo)
#> 
#> FALSE  TRUE 
#>   235    91

occ_thin_geo <- remove_flagged(occ = occ_clean_thin)

heatmap_thin_geo <- spatial_kde(occ = occ_thin_geo, resolution = 0.2, 
                                buffer_extent = 50, radius = 2, 
                                zero_as_NA = TRUE)

ggmap_here(occ_clean_thin, size_points = 0.5, heatmap = heatmap_thin_geo)

6.3. Escolhendo uma distância de thinning

Mas como escolher uma distância apropriada? Essa decisão costuma ser arbitrária.

O RuHere implementa uma abordagem baseada em Moran’s I, permitindo testar múltiplas distâncias e selecionar uma configuração que reduza autocorrelação espacial.

data("worldclim", package = "RuHere")
r <- terra::unwrap(worldclim)

occ_thin_moran <- flag_geo_moran(
  occ = occ_clean, 
  d = c(1, 3, 5, 7, 10, 15, 20, 30),
  env_layers = r
)

occ_thin_geo_moran <- remove_flagged(occ = occ_thin_moran$occ)

heatmap_thin_geo_moran <- spatial_kde(occ = occ_thin_geo_moran, 
                                      resolution = 0.2, 
                                      buffer_extent = 50, radius = 2,
                                      zero_as_NA = TRUE)

ggmap_here(occ = occ_thin_moran$occ, size_points = 0.5, heatmap = heatmap_thin_geo_moran)

Na prática, essa função ajuda a evitar escolhas arbitrárias de distância para o thinning.

6.4. Thinning ambiental

Um possível problema do thinning geográfico é que registros próximos no espaço podem, na verdade, representar condições ambientais muito diferentes, especialmente em regiões ambientalmente heterogêneas.

Nesses casos, remover registros apenas pela proximidade geográfica pode levar à perda de informações importantes sobre o nicho ecológico da espécie.

Ideia principal: dois registros podem estar próximos geograficamente, mas ocorrer sob climas bastante diferentes.

Para contornar esse problema, o RuHere também permite realizar thinning no espaço ambiental.

Em vez de considerar apenas distância geográfica, essa abordagem remove registros que representam condições ambientais redundantes.

O procedimento funciona criando uma grade multidimensional no espaço ambiental, dividindo cada variável em um número definido de intervalos (n_bins). Registros que caem no mesmo bloco ambiental são considerados muito semelhantes e podem ser marcados como redundantes.

6.4.1. Entendendo os blocos ambientais

Para visualizar como essa discretização funciona, podemos criar os blocos ambientais usando get_env_bins().

b <- get_env_bins(

6.4.2. Aplicando o thinning ambiental

Agora podemos marcar registros redundantes usando thin_env().

Neste exemplo, usamos n_bins = 10 e priorizamos registros mais recentes usando o ano de coleta.

  occ = occ_thin_moran$occ,
  env_layers = r,
  n_bins = 10,
  prioritary_column = "year"
)

#>   255    71

Depois, podemos remover os registros marcados e visualizar o padrão espacial resultante.

occ_thinned_env <- remove_flagged(
  occ = occ_thin_env
)

heatmap_thin_env <- spatial_kde(
  occ = occ_thinned_env,
  resolution = 0.2,
  buffer_extent = 50,
  radius = 2
)

ggmap_here(
  occ = occ_thin_env,
  size_points = 0.5,
  heatmap = heatmap_thin_env
)

O padrão espacial após o thinning ambiental pode ser bastante diferente daquele obtido apenas com distância geográfica.

6.4.3. Escolhendo o número de bins

Assim como no thinning geográfico precisamos escolher uma distância, no thinning ambiental precisamos definir o número de blocos (n_bins).

Valores maiores geram uma divisão mais refinada do espaço ambiental, reduzindo a chance de registros serem considerados redundantes.

O RuHere permite testar diferentes valores automaticamente usando Moran’s I, buscando reduzir autocorrelação espacial sem remover registros excessivamente.

occ_env_moran <- flag_env_moran(

  n_bins = c(5, 10, 20, 30, 40, 50, 60, 70, 80),
  prioritary_column = "year",

6.4.4. Combinando thinning geográfico e ambiental

As duas abordagens possuem vantagens e limitações.

O thinning geográfico pode remover registros ambientalmente únicos. Já o thinning ambiental pode eliminar registros geograficamente importantes.

Por isso, uma estratégia interessante é combinar ambas usando flag_consensus().

occ_thinned <- flag_consensus(
  occ = occ_env_moran$occ,
  flags = c(
    "thin_geo",
    "thin_env"
  ),
  consensus_rule = "any_true",
  flag_name = "thin_geo_env"
)

Depois, podemos visualizar os registros marcados:

ggmap_here(
  occ = occ_thinned
)

Na prática, combinar critérios geográficos e ambientais pode ajudar a equilibrar representatividade espacial e diversidade ambiental.

Considerações finais

Ao longo desta aula, percorremos um workflow completo para:

  1. obter registros de ocorrência;
  2. padronizar e integrar diferentes bases;
  3. identificar inconsistências espaciais;
  4. validar registros usando informações de especialistas;
  5. visualizar e remover registros problemáticos;
  6. reduzir viés amostral espacial e ambiental.

O RuHere busca oferecer um fluxo de trabalho:

  • reproduzível;
  • transparente;
  • flexível;
  • centrado na inspeção iterativa dos dados.

Mais do que remover registros automaticamente, a proposta do pacote é auxiliar pesquisadores a tomarem decisões mais informadas durante a preparação de dados de ocorrência.

Mensagem principal: dados de ocorrência raramente vêm prontos para análise.

Investir tempo em validação e inspeção geralmente melhora substancialmente a qualidade das inferências ecológicas e biogeográficas.
